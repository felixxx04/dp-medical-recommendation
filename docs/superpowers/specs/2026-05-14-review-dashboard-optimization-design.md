# 推荐审核页面优化设计

## 背景

ReviewDashboard 页面存在两个核心问题：
1. **视觉不美观**：pending 列表用 inline style，审核区缺乏层次感，与现代极简风格差距大
2. **滚动体验差**：点击下方待审核项后，审核详情区在视口上方，必须手动滚回

## 设计决策

- **视觉风格**：现代极简风（Notion/Linear 式），大留白、清晰分层、徽标色彩
- **布局方案**：左右分栏 + 右侧 sticky，彻底解决滚动问题
- **交互优化**：提交审核后自动选中下一条，支持流畅批量审核

├──────────────┬──────────────────────────────┤
│  待审核列表   │  审核详情（sticky）           │
│  (滚动区)    │                              │
│              │  ┌────────────────────────┐  │
│  ┌────────┐  │  │ 层1: 患者信息           │  │
│  │ ● 高血压│  │  ├────────────────────────┤  │
│  └────────┘  │  │ 层2: 推荐药物列表       │  │
│  ┌────────┐  │  ├────────────────────────┤  │
│  │ ○ 糖尿病│  │  │ 层3: 审核操作           │  │

## 左侧：待审核列表

- **紧凑卡片**：每项一行，左侧 3px 色条标识状态
  - 待审核：橙色色条
  - 已审核：灰色细条
- **选中态**：背景微灰 + 左侧色条加粗（4px），区别于 hover 态
- **hover 态**：背景极浅灰
- **已审核历史**：列表下方可折叠区块，点击展开/收起，已审核项只读
- **空状态**：居中文案 "暂无待审核记录"

### 列表项结构

```
[色条] 疾病名称              时间
       患者姓名 · 推荐药物数
```

## 右侧：审核详情（sticky）

`position: sticky; top: 1rem;` 固定在视口内。

高度限制：`max-height: calc(100vh - 2rem); overflow-y: auto;`，内容过长时右侧内部可滚动。

### 层1：患者信息卡

- 疾病名称（text-xl font-semibold）
- 患者基本信息行：年龄 · 性别 · 过敏史
- 用药史标签组：小 pill 标签，浅色背景

### 层2：推荐药物列表

- 每个药物一行：
  - 药名（左侧）
  - 得分 badge（中间，圆角背景）
  - 安全标签（右侧，绿/黄/红色文字）
- 安全标签色：安全=绿色，需关注=黄色，禁忌=红色

### 层3：审核操作

- 三个按钮横排：确认（绿色轮廓）/ 修改（蓝色轮廓）/ 拒绝（红色轮廓）
- 选择修改时：展开药物多选器
- 选择修改/拒绝时：展开备注 textarea
- 底部"提交审核"实心按钮（品牌色）
- **提交后行为**：
  1. 该条从待审核移到已审核列表
  2. 自动选中下一条待审核项
  3. 若无下一条，显示空状态

## 交互流程

1. 用户点击左侧待审核项 → 右侧详情更新（无需滚动，sticky 已固定）
2. 用户查看详情，选择确认/修改/拒绝
3. 填写必要信息后点击提交
4. 系统提交审核 → 列表更新 → 自动选中下一条
5. 重复直到待审核列表清空

## 文件变更范围

| 文件 | 变更 |
|------|------|
| `src/pages/ReviewDashboard.tsx` | 重构布局：sticky 右侧、列表样式、折叠已审核、提交后自动选中 |

| `src/lib/statusConstants.ts` | 无变更，复用现有状态配置 |

## 不做的事

- 不改变 API 接口
- 不改变审核逻辑（确认/修改/拒绝的语义不变）
- 不引入新的 UI 库
- 不改变路由结构
